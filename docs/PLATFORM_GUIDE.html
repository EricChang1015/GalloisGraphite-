Mada Graphite 平台總覽與操作手冊

B2B 石墨交易平台 — 公開官網、Buyer / Seller 交易流程與 Admin 後台完整說明

本文件依專案實際程式碼（src/app、src/actions、src/proxy.ts）與設計文件 （docs/PRD.md、docs/ARCHITECTURE.md）整理。路由群組 (public)、(auth)、(app) 不會出現在實際 URL 中。按鈕與分頁名稱保留英文原文，方便對照實際介面。

1. 專案概覽 ↑ 目錄

Mada Graphite 是一個 B2B 石墨交易平台，連接全球買家與馬達加斯加在地賣家， 主要供應方為自有礦場的 Graphite Energy Inc.。平台整合 AI 智能諮詢、市場情報與線上 B2B 交易， 涵蓋從詢價、議價、合約簽署、多階段付款到出貨清關的完整流程。

核心交易流程

議價發生在 inquiry / quotation 層；買家接受報價後直接建立訂單並進入 contract_pending。 付款流程與訂單狀態解耦，由 payment_schedules 平行管理，未付清的分期只影響能否 completed，不卡其餘進度。

[詢價議價 — inquiry 層]
 Buyer 提交 Inquiry ──► Seller 報價 Quotation ──► 雙方 Counter 來回 ──► Buyer Accept
                                                                        │
[訂單 — order，12 階段線性]                                          ▼
 contract_pending ─► contract_signed ─► in_production ─► ready_to_ship
   ─► shipped ─► in_transit ─► arrived ─► customs_cleared ─► completed
   （任一節點可 ─► disputed / cancelled）
                                                                        ▲
[付款 — payment_schedules，平行]                                    │
 scheduled ─► due ─► awaiting_review ─► paid  ───（全部 paid 才能完成）──┘

3. 帳號與通用功能 ↑ 目錄

3.1 註冊與登入

  • Email 註冊（/register）：填 email、密碼、角色（buyer / seller）、公司名 / 國家。系統寄出驗證信，點擊連結後帳號 status 由 pending 轉 active。
  • Email 驗證（/verify）：可重寄驗證信。
  • Google OAuth：一鍵登入，預設角色 buyer，公司資料可日後補齊。
  • 忘記 / 重設密碼（/forgot-password → /reset-password）：寄 recovery 連結；Google 用戶可藉此額外綁定 email/password，同一 profile 兩種方式皆可登入。

3.2 共用儀表板與設定

路由功能
/dashboard角色感知首頁：依角色顯示待辦（待回覆詢價、待付款、待簽合約）、進行中訂單與快速連結；admin 角色另顯示「Go to Admin Console」。
/settings個人 / 公司資料、頭像、語言切換、KYC 入口。company_name 與 country 為商業資料 gate（詳下）。
/settings/kycKYC：手機驗證、證件上傳，四級 level（0–3）。
/messages · /messages/[userId]站內訊息：每對 buyer + seller 一條 thread，使用 Supabase Realtime；可從 market、inquiry、order 頁的「Message」按鈕開啟。
商業資料 Gate：當 profiles.company_name 或 country 為空時， createInquiry（買家）、createListing（賣家）、submitPayment 會回傳 PROFILE_INCOMPLETE，前端以 toast 提示並導向 /settings?prompt=incomplete 補齊。
i18n：儀表板（(app)/**）與核心公開行銷頁支援英文 / 簡體中文（cookie mg-locale + profiles.locale）； 合約正文、email、SMS、admin 後台、news 與 chat 內容目前仍為英文。

4. Buyer 買家端 ↑ 目錄

buyer 買家的核心任務是「找貨 → 詢價 → 議價 → 簽約 → 分期付款 → 收貨」。以下依交易階段說明功能與逐步操作； 每一步標註對應路由與主要 UI 元件。

4.1 瀏覽與詢價

功能介紹：買家在市場瀏覽 active listing，進入單品詳情查看規格、圖庫與賣家資訊，並送出詢價（Inquiry）。

  1. 瀏覽市場
    進入 /market，以卡片瀏覽所有上架商品，卡片顯示 spec chip、最小起訂量（MOQ）與縮圖。
  2. 查看單品
    點入 /market/[id]，透過 <ListingGallery /> 看 hero + 縮圖、規格、賣家與可出貨時間。
  3. 送出詢價
    點「Submit Inquiry」開啟 <InquiryDialog />，填 requested_qty、target_price、destination、message。系統檢查 MOQ、商業資料與 KYC 門檻，通過後建立 inquiry。
  4. 追蹤詢價
    在 /inquiries 的 Sent 分頁查看已送出詢價的狀態。
  5. 與賣家溝通
    可用單品頁的「Message」按鈕開啟 /messages/[sellerUserId] 私訊賣家。

4.2 議價與下單

功能介紹：賣家回覆 quotation 後，買家可在詢價詳情頁審閱報價、議價或接受。接受報價即自動建立訂單。

  1. 審閱報價
    進入 /inquiries/[id]，以 <QuotationActions /> 看 quotation timeline（規格 / 價格 / Incoterm / 有效期）。
  2. 議價或接受
    可選 Counter（counterQuotation，inquiry 轉 negotiating）、Decline，或 Accept。
  3. 接受報價建立訂單
    acceptQuotation 會建立 order（status = contract_pending），把 Incoterm 快照到訂單，inquiry 轉 converted，並導向 /orders/[id]。
提案方不能接受自己提出的 offer；接受動作一律由收到報價的一方執行。

4.3 合約簽署

功能介紹：賣家起草合約後，買家在 Contract 分頁預覽、核准或退回，並上傳簽名掃描檔。

  1. 預覽合約
    在 /orders/[id] 的 Contract 分頁，以 <ContractPreview /> 檢視合約內容與付款排程表，可用 window.print() 輸出 PDF。
  2. 核准或退回
    <ContractApproveReject /> 提供 Approve（approveContract）與 Reject（rejectContract，附 reason → 賣家重新起草，revision_no++）。
  3. 上傳簽名掃描
    印出簽名後，用 <SignedScanUploader /> 上傳買方簽名件（image 或 PDF）。
  4. 完成簽約
    雙方皆上傳簽名 + 買家 approve 後，訂單自動 contract_pending → contract_signed → in_production。

4.4 付款

功能介紹：付款為平行流程，由 payment_schedules 多筆分期構成，不卡訂單其他進度。賣家為付款主審。

  1. 查看排程
    在 Payment 分頁，<PaymentScheduleTable /> 顯示各分期狀態；進度條右上顯示「Payments X/Y paid」。
  2. 提交付款
    對 due / overdue 分期，用 <PaymentForm /> 提交（submitPayment）；對 scheduled 列亦可「Pay Early」提前結算。
  3. 付款方式
    支援 USDT（TRC20 / ERC20）、USDI、MUP、bank_transfer。鏈上方式填 tx_hash；銀行 / USDI / MUP 上傳付款憑證圖到 Storage。
  4. 等待審核
    提交後分期轉 awaiting_review，由賣家 verify；若被 reject 則回 due，需重新提交。

4.5 出貨追蹤與收尾

  1. 追蹤物流
    Shipment 分頁查看 B/L No、vessel、container、ETD / ETA / ATA。
  2. 文件中心
    Documents 分頁查看 / 上傳 13 種文件（Contract / Invoice / Logistics / Inspection / Customs / Payment / Other）。
  3. 到港與清關
    任一方可 markArrived（記 ATA）；買家 markCustomsCleared 進入 customs_cleared。
  4. 完成訂單
    清關時若全部 schedule 已 paid，自動 completed；否則停在 customs_cleared，待最後一筆付款 verify 後自動完成。

5. Seller 賣家端 ↑ 目錄

seller 賣家的核心任務是「上貨 → 回應詢價 → 起草合約 → 生產出貨 → 審核付款」。

5.1 上貨管理

功能介紹：賣家建立並維護自己的 listing，含結構化規格、MOQ、圖片與上下架狀態。

  1. 建立 listing
    進入 /listings/new，選品類（讀 product_categories.spec_schema）、填 title、specs、數量 / 單位、出貨地、可出貨時間區間、單價 / 幣別、Incoterm、備註與圖片。createListing 後立即出現在 market。
  2. 規格與 MOQ
    支援結構化 Flake Graphite 規格（product_type / mesh_size / fixed_carbon 範圍等）與 Custom Grade 自填 mesh；可設 optional MOQ，並用「Generate title」自動生成標題。
  3. 上傳圖片
    <ListingImageUploader /> 支援上傳新圖或重用既往圖庫，上傳前自動壓到 720px WebP。
  4. 管理 listing
    在 /listings 的 Actions 欄行內操作：Edit（/listings/[id]/edit）、Pause / Resume、Sold-out、Delete（已有訂單的 listing 無法刪除，回 LISTING_HAS_ORDERS）。

5.2 報價與議價

  1. 收到詢價
    在 /inquiries 的 Received 分頁查看，並收到 Email / SMS 通知。
  2. 送出報價
    進 /inquiries/[id]，用 <QuotationForm />（submitQuotation）發報價，可預設 listing 條件與 14 天有效期；或對 pending 詢價用 acceptInquiry 快速生成預設報價。
  3. 議價來回
    counterQuotation 送出新報價；rejectInquiry / rejectQuotation 拒絕。
  4. 等待買家接受
    買家 acceptQuotation 後，訂單自動建立並進入 contract_pending。

5.3 合約起草

  1. 起草合約
    在訂單 Contract 分頁，用 <ContractDraftForm /> 選 Incoterm（FOB / CFR / CIF），由模板動態生成 HTML。
  2. 設定付款排程
    <PaymentScheduleBuilder /> 拆出 1–10 筆分期（prepayment / regular / postpayment，總和須為 100%），建立 payment_schedules。
  3. 上傳簽名掃描
    印出簽名後用 <SignedScanUploader /> 上傳賣方簽名件。
  4. 買家退回時
    若買家 reject，可重新 draftContract（revision_no 遞增）。

5.4 生產與出貨

  1. 生產中
    簽約後通常自動 in_production；可用 <OrderPhaseActions /> 操作各階段。
  2. 備貨完成
    markReadyToShip → ready_to_ship。
  3. 出貨
    Shipment 分頁用 <ShipmentForm /> 填 B/L No、vessel name / IMO、container、ETD / ATD / ETA，markShipped → shipped；可選上傳 B/L 掃描與 Inspection Report（COA / SGS）。
  4. 在途與到港
    markInTransit → in_transit；任一方 markArrived → arrived。

5.5 付款審核（賣家為主審）

  1. 審核付款
    買家提交付款後分期轉 awaiting_review，賣家在 Payment 分頁用 <PaymentVerifyActions /> 執行 verifyPayment：通過 → schedule paid；拒絕 → 回 due。
  2. Milestone 手動觸發
    <MilestoneActionButtons /> 觸發細節點：賣家側如 markBeforeProduction / markBeforeShipment / markBeforeLoading；買家側如 markBlReceived / markGoodsPickedUp 等（依 Incoterm 顯示）。粗節點隨訂單狀態自動觸發，bl_date_plus_N 由 cron /api/cron/payment-schedule 排程。
付款審核權設計：賣家為 primary reviewer，Admin 在 /admin/payments 保留覆審 / 爭議介入權（migration 015）。

6. Admin 後台 ↑ 目錄

admin super_admin 後台位於 /admin/*，僅 admin 或 super_admin 可進入。導覽包含 10 條路由， 涵蓋平台運維、交易監管與內容管理。Admin actions 分散於 src/actions/admin.ts、 admin-news.ts、admin-mine-photos*.ts、admin-partners*.ts 與跨域的 order.ts / payment.ts / document.ts。幾乎所有 admin mutation 都寫入 audit_logs。

6.1 儀表板總覽

路由：/admin

  • 統計卡：總用戶數、總訂單數、爭議訂單數、待審付款數。
  • Priority Actions：爭議訂單、待審付款、需處理用戶（KYC pending / status pending / profile 不完整）。
  • 側邊欄 badge：Users（金）、Payments（金）、Orders disputed（紅），即時提醒待辦。

6.2 用戶與 KYC 管理

路由：/admin/users

  1. 瀏覽與篩選
    查看全平台 profiles（email、公司、角色、狀態、KYC level），可依 attention / role / status / KYC 篩選與搜尋。
  2. 凍結 / 解凍
    freezeUser（status → frozen）/ unfreezeUser（→ active）。凍結用戶會被 proxy 導回 dashboard。
  3. 變更角色
    setUserRole：buyer / seller / admin。將用戶 promote 為 admin 僅 super_admin 可執行。
  4. KYC 審核
    透過 <UserKycDialog /> 檢視上傳證件（signed URL）與電話驗證狀態，approveKycDocuments 核准後自動升級 level，或用 setUserKycLevel 手動設定（0–3，含備註）。

6.3 訂單、付款與文件監管

路由能做什麼
/admin/orders 全平台訂單列表（order_no、buyer / seller、金額、狀態）。
/admin/orders/[id] 訂單詳情 + Admin 介入：進度條、合約簽名狀態、付款排程、文件中心、timeline；顯示該訂單 audit log（最近 20 筆）。 <AdminOrderActions /> 提供 Force Transition（繞過狀態機強制切換，爭議解決 / 恢復用）、 raiseDispute、cancelOrder（出貨前；會 waive 未付 schedule、reject pending payments）。
/admin/payments 付款覆審：Pending Review + History。與賣家共用 <PaymentVerifyActions /> 執行 verify / reject， 作為賣家失職 / 爭議時的 fallback 審核。買家提交付款時 admin 會收到 CC email。

文件核驗：Admin 可在訂單詳情標記 order_documents 為已核驗（verifyOrderDocument）並刪除任何文件（deleteOrderDocument）；當事人僅能在上傳後 1 小時內刪除未驗證文件。

6.4 內容管理（CMS）與平台設定

路由能做什麼
/admin/categories產品品類 CRUD 與結構化 spec_schema（product_type / mesh_size / fixed_carbon / moisture 等）；軟刪除（is_active=false）/ 重新啟用。
/admin/news新聞 CMS：tab 分 pending / published / rejected / draft。手動 CRUD（upsertNews）、自動抓取候選（fetchNewsCandidatesAction / importNewsCandidatesAction）、審核（approve / reject / unpublish）與多語翻譯（translateNewsArticleAction）。
/admin/mine-photos礦場照片 CMS：分類 CRUD（slug / 中英標題 / 排序 / 發布）、上傳 / 刪除照片、設定分類封面。
/admin/partners首頁合作夥伴管理：partner CRUD（slug / name / href / 排序 / 發布）與 logo 上傳。
/admin/settings平台設定：SMTP 狀態與「Send test email」（驗證 AWS SES）、KYC 門檻（詢價 / 上架最低 level）、SMS 通知開關。
後台刻意不含的功能：沒有全域 audit log 瀏覽頁（僅 per-order 片段可見）、 沒有專屬 chat 管理後台（admin 可在資料層讀取任何 chat room，但無發訊 UI）、 AI chat log 僅 DB RLS 可查無 UI、無自動新聞爬蟲（改半自動抓取審核）、無區塊鏈 / 自動結算。

7.2 訂單狀態機（12 階段線性）

quotation_pending ─► quoted ◄─► negotiating
   └─ buyer accept quotation
        └─ contract_pending ◄── 賣家可重新起草（revision_no++）
             └─ buyer approve + 雙方上傳簽名掃描
                  └─ contract_signed ─► in_production ─► ready_to_ship
                       ─► shipped ─► in_transit ─► arrived
                       ─► customs_cleared ─► completed

任一非終止狀態 ─► disputed / cancelled
定義：src/lib/order/stateMachine.ts
實務上議價在 inquiry 層（pending → quoted ↔ negotiating → converted）完成， acceptQuotation 建立 order 時狀態已是 contract_pending。

7.3 付款排程生命週期

scheduled ─► due ─► awaiting_review ─► paid
   │            │
   │            └─► overdue（逾期）
   └─► waived（取消訂單時 waive）

買家 submitPayment → awaiting_review；賣家 / admin verifyPayment → paid（拒絕回 due）
1–10 筆 installment / 訂單，總和 100%；bl_date_plus_N 由 cron 觸發
